# English Grammar Guide: Modal Verbs 🛠️💡

Welcome to this comprehensive guide on **Modal Verbs**! Modals are special helping verbs (like *can, should, must*) that change or add meaning to the main verb. They express ideas like ability, permission, advice, and obligation.

---

## 1. The Core Rules of Modals 🌟

Before jumping into the types of modal verbs, there are three absolute grammar rules you must follow:

1.  **No "-s" for Singular:** Modals never change their form. They look exactly the same for *I, you, he, she, they*.
    *   ❌ *Incorrect:* He **cans** code in HTML.
    *   ✅ **Correct:** He **can** code in HTML.
2.  **Followed by Base Verbs:** Always use the bare infinitive (the base verb *without* "to") immediately after a modal.
    *   ❌ *Incorrect:* You should **to study** tonight.
    *   ✅ **Correct:** You should **study** tonight.
3.  **No "Don't / Doesn't" in Negatives:** To make a modal sentence negative, simply add **not** directly to the modal verb.
    *   ❌ *Incorrect:* You don't must paste that code.
    *   ✅ **Correct:** You **must not** / **mustn't** paste that code.

---

## 2. Categorizing Modals by Function

We use different modal verbs depending on what message we want to send:

### A. Ability (What you are able to do)
*   **Can:** Used for present or general ability.
    *   *Example:* Celine **can** build beautiful websites on GitHub.
*   **Could:** Used for past ability.
    *   *Example:* Yesterday, I **couldn't** find where the loading glitch was.

### B. Permission & Possibility (Asking to do something / Guessing)
*   **Can / May / Could:** Used to ask for or give permission. (*May* and *Could* are more polite than *Can*).
    *   *Example:* **May** I open a new issue on your repository?
*   **Might / May / Could:** Used when something is possible, but you aren't 100% sure.
    *   *Example:* It **might** rain later, so keep your iPad safe.

### C. Advice & Suggestions (Giving recommendations)
*   **Should / Ought to:** Used to give good advice or state what is best to do.
    *   *Example:* You **should** use a unique filename so GitHub doesn't give you an error.

### D. Obligation & Necessity (Rules and requirements)
*   **Must / Have to:** Used when something is a strict rule or absolutely mandatory.
    *   *Example:* You **must** commit your changes to save your code file.
*   **Must not (Mustn't):** Used for things that are completely forbidden.
    *   *Example:* You **must not** share your GitHub password with anyone.

## 3. Quick Reference Summary Table 📊

| Modal | Main Usage | Quick Example |
| :--- | :--- | :--- |
| **Can** | Present Ability / Informal Request | I **can** finish my homework early. |
| **Could** | Past Ability / Polite Request | **Could** you help me fix this branch? |
| **Should** | Advice / Recommendation | We **should** test the HTML layout first. |
| **Must** | Strict Obligation / Core Rule | Everyone **must** follow the repository guidelines. |
| **May** | Formal Permission | **May** I sit here? |
